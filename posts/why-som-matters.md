The web has three eras of consumption:

1. **Browsers** read HTML and render pixels
2. **Search engines** read HTML and extract links and text
3. **AI agents** read HTML and... struggle

Every prior consumer of the web got a format designed for them. Browsers got HTML, CSS, and JavaScript. Search engines got sitemaps, robots.txt, and structured data (Schema.org, JSON-LD). AI agents got nothing. They're still reading raw HTML and hoping for the best.

That's what SOM fixes.

## What's wrong with HTML for agents?

HTML is a rendering format. It encodes how a page *looks*, not what it *means*. Consider a product page:

- The price might be in a `<span class="a-price-whole">` (Amazon)
- Or a `<div data-testid="price-display">` (Target)
- Or a `<meta itemprop="price" content="29.99">` (Schema.org)
- Or generated by JavaScript after page load

An agent reading raw HTML has to figure out which of the 500+ elements on the page is the price. That's expensive (tokens), slow (reasoning time), and unreliable (different sites use different patterns).

## What's wrong with Markdown for agents?

Markdown is better -- it strips the noise. But it also strips the meaning:

```markdown
Sign Up Free
```

Is that a button? A heading? Decorative text? Markdown doesn't tell you. An agent that needs to *click* "Sign Up Free" can't do that from Markdown alone.

Markdown is fine for reading. It fails for acting.

## What SOM provides

SOM preserves exactly what an agent needs:

### 1. Semantic regions

Every page is divided into regions with explicit roles: `navigation`, `main`, `header`, `footer`, `form`. An agent looking for the main content doesn't need to scan 500 elements -- it goes straight to the `main` region.

### 2. Typed elements

Every element declares what it is: `link`, `button`, `text_input`, `select`, `heading`, `paragraph`. No guessing.

### 3. Available actions

Interactive elements include their affordances: `["click"]`, `["type", "clear"]`, `["toggle"]`, `["select"]`. An agent knows exactly what it can do.

### 4. Stable IDs

Every element gets a deterministic ID derived from its content and position. The same button produces the same ID across page loads, enabling agents to reference elements across sessions.

### 5. HTML ID bridge

When the source element has an HTML `id` attribute, SOM preserves it as `html_id` -- giving agents a direct bridge back to the DOM for interaction via selectors or JavaScript.

### 6. ARIA state awareness

SOM captures ARIA states (`expanded`, `selected`, `checked`, `disabled`) so agents understand dynamic widget state without needing to execute JavaScript.

## The practical impact

From our [WebTaskBench benchmark](https://blog.plasmate.app/html-vs-markdown-vs-som/):

- **4.0x fewer tokens** than raw HTML
- **Faster inference** than both HTML and Markdown on GPT-4o and Claude
- **5.4x compression on navigation tasks** -- the category where agents interact most
- **6.0x compression on adversarial pages** -- heavy JavaScript, ads, cookie banners

SOM doesn't just save tokens. It makes agents faster and more reliable by giving them the right level of abstraction.

## The path forward

We're building SOM as an open standard, not a proprietary format:

- [SOM Spec v1.0](https://docs.plasmate.app/som-spec) -- the full specification
- [SOM-first websites](https://docs.plasmate.app/som-first-sites) -- publishers serving SOM natively via `/.well-known/som.json`
- [robots.txt proposal](https://docs.plasmate.app/robots-txt-proposal) -- cooperative content negotiation for agents
- [W3C Community Group](https://www.w3.org/community/web-content-browser-ai/) -- working toward web standards

The web wasn't built for machines. SOM is the layer that changes that.
